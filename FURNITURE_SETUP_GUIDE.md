# Furniture Data Pre-Loading System\n\n## Overview\nThis system ensures house interior data (furniture locations, positions, colors) is loaded BEFORE any pet action that requires furniture (feed, drink, toilet, shower, sleep).\n\nWhen you click \"Autofarm\" or manually trigger a furniture action, the system:\n1. Fires the house data signal to DataAPI/DataChanged\n2. Waits for it to process\n3. Then invokes the furniture action\n\n## Setup\n\n### Step 1: Get Your House Data\nUse Cobalt to generate your house_interior data:\n```\nlocal Event = game:GetService(\"ReplicatedStorage\").API[\"DataAPI/DataChanged\"]\nfiresignal(Event.OnClientEvent, \n    \"YourPlayerName\",\n    \"house_interior\",\n    {house_pos = ..., furniture = {...}, ...},\n    timestamp\n)\n```\n\nCopy the DATA table (3rd parameter to firesignal).\n\n### Step 2: Initialize Before UI\nBefore calling UI.Init(), initialize the house data:\n\n```lua\nlocal HouseDataSetup = require(script.Parent:FindFirstChild(\"HouseDataSetup\"))\n\nlocal myHouseData = {\n    house_pos = Vector3.new(-6000, 4000, -9000),\n    furniture = {\n        [\"f-2\"] = {...},\n        [\"f-6\"] = {...},\n        [\"f-8\"] = {...},\n        -- etc\n    },\n    unique = \"house_{...}\",\n    -- other fields\n}\n\nHouseDataSetup.loadFromCobalt(myHouseData)\n\n-- Now call UI.Init with optional 8th parameter:\nUI.Init(Pets, Sleep, Care, Remotes, PetState, Toys, Requirements, myHouseData)\n```\n\n### Step 3: That's It!\nWhenever you:\n- Click \"Autofarm\"\n- Click manual actions (Feed, Drink, etc.)\n- Call any furniture action\n\nThe house data is automatically fired to DataAPI/DataChanged first, ensuring it's available.\n\n## Files\n\n### Core Files\n- **HouseData.lua** - Stores cached data and fires the signal\n- **HouseDataSetup.lua** - Helper to initialize from Cobalt data\n- **ui-refactored.lua** - Modified to call HouseData.ensureLoaded() before actions\n\n### Reference\n- **EXAMPLE_WITH_HOUSE_DATA.lua** - Complete working example\n- **HOUSE_DATA_SETUP.md** - Detailed guide\n\n## How It Works\n\n### Flow\n```\nUser clicks \"Feed\" or \"Autofarm\"\n    ↓\ninvokeHardcodedFurnitureAction()\n    ↓\nHouseData.ensureLoaded(playerName)\n    ↓\nfiresignal(DataChanged.OnClientEvent, playerName, \"house_interior\", cachedData, timestamp)\n    ↓\nDataAPI processes signal\n    ↓\nActivateFurniture RemoteEvent fires\n    ↓\nPet performs action\n```\n\n### Throttling\n- Data is cached after first load\n- Signal fires max once per 0.5 seconds (throttled to prevent spam)\n- Works for any player name (dynamic)\n\n## Usage Patterns\n\n### Pattern 1: Static Data\nYou have Cobalt output and it never changes:\n```lua\nHouseDataSetup.loadFromCobalt(staticHouseData)\n-- Data stays loaded for entire session\n```\n\n### Pattern 2: Dynamic Updates\nYou get new furniture data later:\n```lua\nHouseDataSetup.loadFromCobalt(newHouseData)\n-- Next furniture action uses updated data\n```\n\n### Pattern 3: Multiple Players\nOther players use the same script:\n```lua\n-- Each player passes their own house data\nUI.Init(Pets, Sleep, Care, Remotes, PetState, Toys, Requirements, playerAHouseData)\n```\n\n## Troubleshooting\n\n### \"Signals: awaiting ailments_manager\"\n- This is about pet ailments, not house data\n- House data system fires separately on furniture actions\n\n### Actions still failing\n- Verify house data has correct furniture IDs (f-2, f-6, f-8, etc.)\n- Check that CFrame data matches actual positions\n- Ensure \"furniture\" table is present in data\n\n### Data not updating\n- Call HouseDataSetup.loadFromCobalt(newData) again\n- Old cached data is replaced\n\n## Performance\n- Throttling prevents signal spam\n- Cached data = no repeated network calls\n- Fire signal only on furniture actions, not idle\n"
